프롬프트 엔지니어링의 정의와 한계

정의
모델로부터 원하는 답변을 얻기 위해 질문을 구체적으로 설계하는 기술
더 구체적인 질문일수록, 더 유용한 답변을 이끌어낼 수 있습니다.
✕ 일반 프롬프트
"좋은 마케팅 전략 알려줘"
모호하고 단편적인 답변 유도
›
✓ 구체적 프롬프트
"대학생 대상 스타트업, 예산 월 100만 원, 온라인 홍보만 가능. 마케팅 전략을 짜줘"
상황을 반영한 실질적 답변 유도
⚠ 한계

CONTEXT ENGINEERING

콘텍스트 엔지니어링의 3원칙

⬇
01
시스템적으로
접근하기
질문을 잘 쓰는 법이 아니라, 정보를 준비하고 다듬는 공정 전체를 설계하는 일. 의도 파악 → 데이터 수집 → 요약 → 순서 배열의 흐름으로 모델이 추측 없이 답할 수 있게 만든다.
⇄
02
동적으로
만들기
모든 요청에 같은 정보를 넣지 않는다. 일정 확인엔 캘린더, 기술 질문엔 문서, 고객 응대엔 대화 기록. 필요한 순간에 필요한 맥락만 선택적으로 제공해 효율과 정확성을 높인다.
{ }
03
적절한 형식
선택하기
무엇을 주느냐 못지않게 어떻게 주느냐가 성능을 좌우한다. 긴 원문보다 핵심 요약, '자유롭게'보다 JSON 형식 요청, 에러 로그도 원인 중심으로 간결히 — 형식이 모델의 이해 속도를 결정한다.

Anthropic의 하네스 엔지니어링 관점

긴 작업을 끝까지 해내는 실행 구조 설계

핵심 정의
'Agent harness는 모델이 에이전트로 행동할 수 있도록 입력을 처리하고, 도구 호출을 조율하고, 결과를 반환하는 시스템' — Anthropic
긴 작업의 구조화
작업이 길어질수록 모델은 맥락을 잃거나, 일을 과도하게 묶거나, 조기 완료를 선언하기 쉽습니다.
· 초기화 에이전트 → 기능 목록 및 진행 기록 정리
· 코딩 에이전트 → 한 번에 한 기능씩 순차 구현
운영 설계의 철학
하네스는 고정된 정답이 아닙니다. 모델이 발전하면 기존 구조는 낡을 수 있습니다.
