RAG 구현 기법 - 청킹 전략

원본 문서를 의미 있는 단위로 분할하는 것은 RAG 시스템의 검색 정확도와 효율성에 직접적인 영향을 미칩니다. 적절한 청킹 전략은 검색기가 관련성 높은 정보를 더 잘 찾아내고, 생성기가 이를 효과적으로 활용하여 답변을 생성할 수 있도록 돕습니다.

고정 크기 청킹

문서를 미리 정해진 고정된 길이(예: 256토큰, 512토큰)로 분할하는 가장 기본적인 방법입니다. 각 청크는 독립적으로 처리됩니다.

구현이 간단하고 예측 가능

문맥이 청크 경계에서 잘릴 수 있음

적합한 사용 사례: 문서의 내용이 비교적 균일하고, 문맥의 연속성이 덜 중요한 경우

고정 크기 + 오버랩

인접한 청크 사이에 일정 부분(예: 10~20%의 토큰)을 중복시켜 문맥 손실을 최소화합니다.

문맥 손실 위험을 줄여 검색 정확도 높임

데이터 중복으로 인해 저장 공간 및 처리량 증가

적합한 사용 사례: 문맥의 연속성이 중요하지만, 복잡한 구조 분석이 어려운 경우

재귀적 분할

문서를 특정 구분자(예: 단락, 문장, 제목)를 기준으로 재귀적으로 분할합니다. 먼저 큰 단위로 나누고, 각 단위를 다시 작은 단위로 나누는 방식입니다.

문서의 계층적 구조를 반영하여 의미 단위로 청크 생성

구현이 복잡하고, 적절한 구분자를 찾는 것이 중요

적합한 사용 사례: 구조화된 문서(예: 보고서, 논문, 매뉴얼)에서 특정 섹션 검색

의미 기반 청킹

문서의 의미적 유사성을 기반으로 청크를 분할합니다. 문장 임베딩을 생성한 후, 유사도 점수가 낮은 지점에서 청크를 나눕니다.

의미적으로 응집된 청크 생성, 검색 관련성 높임

임베딩 모델 성능에 따라 결과 달라짐, 계산 비용 증가

적합한 사용 사례: 문맥의 의미적 연결성이 매우 중요한 경우, 비정형 텍스트
