1. LLM 기초 — 대규모 언어 모델이란?

What is a Large Language Model?

Definition

수십억 개의 파라미터를 가진 Transformer 기반 신경망으로, 방대한 텍스트 데이터로 다음 토큰을 예측하는 방식으로 사전 학습된 언어 모델

핵심 구성 요소

  • Tokenizer: 텍스트를 숫자(토큰) 시퀀스로 변환. 단어·형태소·바이트 단위 분할
  • Embedding Layer: 각 토큰을 고차원 벡터 공간에 매핑하여 의미를 수치화
  • Transformer Blocks: Self-Attention + Feed-Forward 네트워크가 반복 적층
  • Self-Attention: 시퀀스 내 모든 토큰 간의 관계를 병렬로 계산하는 핵심 메커니즘
  • Output Head: 마지막 레이어가 다음 토큰의 확률 분포(Logits)를 출력

2. 사전 학습 vs 파인 튜닝

Pre-training vs Fine-Tuning — Two Distinct Stages

Stage 1

사전 학습 (Pre-Training)

  • 인터넷, 책, 논문 등 수조 개의 토큰으로 자기 지도 학습
  • 목표: 다음 토큰 예측 — 언어 구조·세계 지식을 파라미터에 내재화

  • 결과물: 언어를 이해하지만 지시를 따르지 못하는 "베이스 모델"
  • OpenAI·Meta·Google 등 대형 조직만 현실적으로 수행 가능

3. 왜 파인 튜닝이 필요한가?

Why Fine-Tuning? — Limitations of Base Models

핵심 문제

베이스 모델은 언어를 이해하지만 우리가 원하는 방식으로 행동하지 않는다. 지시를 따르거나, 도메인 전문성을 발휘하거나, 특정 형식으로 응답하는 능력이 부족하다.

베이스 모델의 3대 한계

지시 무시 (No Instruction Following)

"계약서를 요약해줘"라고 해도 지시를 따르지 않고 텍스트를 계속 이어쓰거나 엉뚱한 응답 생성

도메인 전문성 부족

의료·법률·금융 등 특화 지식이 부족하거나, 있어도 환각(Hallucination)이 잦아 신뢰 불가

기업 고유 스타일 미반영

특정 브랜드 어투, JSON 출력 형식, 사내 용어 체계를 프롬프트만으로 완벽히 제어하기 어려움

파인 튜닝 vs 다른 접근법

프롬프트
입력 최적화만. 간단한 형식 교정에는 유효하나 근본 행동 변경 불가. 비용 가장 낮음
RAG
외부 지식 접근을 추가. 최신 정보·사내 문서에 강하나 모델 행동 자체는 변경 불가
파인 튜닝
행동·지식·스타일을 가중치에 직접 주입. 추론 지연 없이 근본 변화 가능
★ 세 방법의 조합이 실무 최강 전략.
프롬프트→RAG→파인 튜닝 순으로 단계 상향
20%+
법률 도메인 특화 파인 튜닝 시 답변 정확도 향상 사례.
범용 모델 대비 압도적인 벤치마크 달성
파인 튜닝이 유일한 해법인 시나리오
  • 특정 회사 브랜드 어투·페르소나를 일관되게 유지해야 할 때
  • 기밀 데이터로 학습해야 해 RAG의 외부 API 호출이 불가할 때
  • 추론 속도가 중요해 검색 지연을 허용할 수 없을 때
  • 모델이 특정 형식(JSON, XML)만 완벽히 출력해야 할 때

5. 지도 학습 파인 튜닝 (SFT)

Supervised Fine-Tuning — Teaching the Model to Follow Instructions

Definition

레이블링된 지시-응답(Instruction-Response) 쌍 데이터를 사용해 베이스 모델이 사람의 지시를 이해하고 따르도록 학습시키는 기법. 파인 튜닝의 가장 기본이 되는 단계.

학습 데이터 형식 예시

Instruction (지시)
아래 의료 기록을 환자가 이해할 수 있는 쉬운 언어로 요약해 주세요: "환자는 좌심실 비대를 동반한 고혈압 3기로 진단됨..."
→ Response: "혈압이 매우 높아서 심장 좌측 벽이 두꺼워진 상태입니다. 혈압약과 생활 습관 개선이 필요합니다."
Instruction (지시)
다음 계약서 조항의 법적 위험을 분석해 주세요: "을은 계약 종료 후 2년간 경쟁사에 취업할 수 없다."
→ Response: "해당 조항은 경업금지 조항으로, 기간 2년과 업종 제한 범위에 따라 무효 판결 가능성이 있습니다..."

6. Full Fine-Tuning과 그 한계

Full Fine-Tuning — When It Works & Why It's Often Overkill

Full Fine-Tuning이란?

  • 사전 학습된 모델의 모든 파라미터(수십억 개)를 새 데이터로 전부 업데이트
  • 이론적으로 특정 태스크에서 최고 수준의 성능 달성 가능
  • 데이터가 충분하고 컴퓨팅 예산이 무제한일 때만 정당화됨
  • 학습 완료 후 전체 모델을 저장해야 해 수십~수백 GB 필요

3가지 치명적 한계

재앙적 망각 (Catastrophic Forgetting)

새 데이터를 학습하면 기존 지식이 덮어씌워짐. 의료 데이터로 튜닝하면 수학 능력이 저하되는 현상

막대한 비용과 시간

7B 모델 Full FT에도 A100 수 장이 수 시간~수일 필요. 70B 모델은 현실적으로 불가

스토리지·배포 부담

태스크마다 전체 모델을 저장해야 함. 10개 태스크 = 10×(수십 GB). 관리 불가

7. PEFT — 파라미터 효율적 파인 튜닝의 등장

Parameter-Efficient Fine-Tuning — The Solution to Full FT's Problems

PEFT의 핵심 아이디어

  • 베이스 모델 가중치 대부분을 완전 고정(Freeze)한 채로 학습 진행
  • 극히 일부 파라미터만 새로 추가하거나 선택적으로 업데이트
  • 학습 가능 파라미터가 전체의 0.01~1% 수준에 불과
  • 그럼에도 불구하고 Full FT에 근접하는 성능을 대부분 달성
  • 재앙적 망각 위험 대폭 감소 — 베이스 지식이 그대로 보존됨

낮은 연산 비용소비자용 GPU(RTX 3090/4090)로 7B~13B 모델 튜닝 가능. 클라우드 비용 수십 배 절감
최소화된 저장 용량어댑터 가중치만 저장 — 수십 GB 대신 10~100MB. 태스크별 교체·관리 용이
재앙적 망각 방지베이스 모델 가중치를 건드리지 않으므로 기존 범용 지식이 그대로 보존
모듈화 및 Hot-swap하나의 베이스 모델 위에 태스크별 어댑터를 교체 가능. 서비스 운용 효율 극대화
핵심 메시지: PEFT는 Full FT의 단점을 해결하면서도 성능을 거의 보존. 특별한 이유가 없다면 항상 PEFT로 시작하는 것이 현재의 업계 표준

8. LoRA — 저차원 적응의 원리

Low-Rank Adaptation — How It Works

LoRA의 동작 원리

  • 원본 가중치 행렬 W(d×k)는 완전 고정(Frozen) 상태 유지
  • 어텐션 레이어(Q, V projection)에 작은 행렬 A(r×k), B(d×r)를 삽입
  • 순전파 시 출력 = Wx + BAx — 두 경로의 결과를 합산
  • B는 초기값 0으로 시작 → 학습 초기에 원본 출력 그대로 유지
  • 학습 완료 후 W에 BA를 더해 병합(Merge) → 추론 지연 Zero
  • 병합 없이 보관 시 태스크별 Hot-swap 운용 가능
핵심 수식
W' = W + B · A
W(d×k) 대신 A(r×k)와 B(d×r) 두 행렬만 학습.
r은 Rank — 보통 4~64의 작은 정수. 행렬 크기 비교: W = d×k개 파라미터, BA = r×(d+k)개.
r≪d,k 이면 파라미터 수가 획기적으로 줄어듦.

LoRA의 핵심 강점

1

메모리 효율: A, B 저차원 행렬만 VRAM에 올림. 7B/13B 모델을 단일 소비자 GPU로 튜닝 현실화

2

빠른 학습: 역전파 시 A, B만 업데이트. 전체 파라미터 대비 학습 속도 현저히 향상

3

유연한 배포: 병합 옵션으로 지연 제로, 또는 분리 보관으로 멀티태스크 Hot-swap

4

범용 호환: Llama, Mistral, Phi, Gemma 등 Transformer 기반 모델 전반에 적용 가능

10. QLoRA — 양자화된 LoRA

Quantized LoRA — 70B+ Models on Consumer Hardware

QLoRA란?

  • 베이스 모델을 4비트 정밀도(NF4)로 양자화한 뒤 LoRA를 적용하는 기법
  • VRAM 사용량을 Full FT 대비 최대 80% 절감
  • 70B+ 초대형 모델을 단일 48GB GPU로 파인 튜닝 가능
  • Full FT에 근접하는 성능을 유지하면서 비용을 혁신
  • Dettmers et al. (2023) 논문 공개 후 빠르게 업계 표준으로 정착

QLoRA 적용 단계

1
베이스 모델을 4-bit NF4로 양자화하여 메모리 적재
↓
2
순전파 시 연산을 위해 BF16으로 역양자화 (Double Quantization 적용)
↓
3
LoRA 어댑터(A, B 행렬)만 BF16 정밀도로 학습
↓
4
학습 완료 후 어댑터 가중치만 저장하여 배포

핵심 양자화 기법

NF4 (4-bit NormalFloat)

가중치 분포를 정규화하여 4비트로 저장. 일반 INT4보다 표현력이 높아 성능 저하 최소화. QLoRA의 핵심 혁신

Double Quantization

양자화 상수 자체를 다시 양자화. 파라미터당 추가로 약 0.37bit 절감하여 VRAM 사용량을 추가로 줄임

LoRA vs QLoRA 선택 기준

  • VRAM ≥ 24GB: LoRA로 시작 — 구현 단순, 성능 우수, 학습 빠름
  • VRAM < 24GB 또는 30B+ 모델: QLoRA 필수 선택
  • 성능 차이: QLoRA는 LoRA 대비 약간 낮지만 실용적으로 무시 가능한 수준
  • 학습 속도: QLoRA는 역양자화 오버헤드로 LoRA보다 10~20% 느림
  • 권장 조합: QLoRA + Unsloth로 속도·메모리를 동시에 최적화
실무 황금률: "QLoRA로 시작하고, 성능 격차가 있을 때만 LoRA → Full FT를 고려하라"

11. 기타 PEFT 기법 — Prefix · Prompt · Adapter

Other PEFT Techniques — Beyond LoRA

LoRA와 QLoRA가 현재 주류이지만, 자원이 극히 제한되거나 여러 태스크를 모듈식으로 관리해야 할 때 유용한 경량 PEFT 기법들

Prompt Tuning

가장 가벼운 기법
  • 입력 임베딩 앞에 학습 가능한 '소프트 프롬프트' 벡터 추가
  • 베이스 모델 파라미터는 완전 고정 — 프롬프트 토큰만 학습
  • 학습 파라미터가 극도로 적어 자원 소모 최소
  • 모델 크기가 클수록 효과 증가 (10B+ 에서 Full FT 수준 가능)
  • 소형 모델(1~3B)에서는 성능 저하 폭이 큼
최적 시나리오: 하나의 거대 모델로 수십 개 태스크를 소프트 프롬프트만 교체해 운용할 때

Prefix Tuning

레이어별 제어
  • 입력 시퀀스 앞에 학습 가능한 프리픽스 토큰 추가
  • Prompt Tuning과 달리 모든 레이어의 어텐션 KV에 주입
  • 각 레이어가 독립적으로 컨텍스트를 제어 — 더 세밀한 조정 가능
  • 연속적인 벡터로 모델의 맥락 파악 능력을 미세 조정
  • Prompt Tuning보다 표현력이 높으나 파라미터가 더 많음
최적 시나리오: 자연어 생성 품질을 섬세하게 조정하면서 베이스 모델을 완전 공유해야 할 때

Adapter

모듈식 관리
  • 트랜스포머 블록 사이에 작은 병목(Bottleneck) 모듈 삽입
  • 기존 가중치는 고정, 어댑터 내부 파라미터만 학습
  • 다운 프로젝션 → 비선형 활성 → 업 프로젝션 구조
  • 태스크별 어댑터 교체가 용이해 모듈화 관리에 최적
  • 단점: 추론 시 어댑터를 통과해야 해 약간의 지연 발생
최적 시나리오: 수십 개의 태스크를 하나의 모델로 서빙하면서 각 태스크 어댑터를 독립적으로 학습·교체할 때

12. 정렬 튜닝 — RLHF와 DPO

Alignment Tuning — RLHF & DPO

Why Alignment?

SFT만으로는 모델이 지시는 따르지만 인간이 진정 원하는 응답을 하지 못함. 정렬 튜닝은 모델이 안전하고 도움이 되며 정직한 방향으로 인간 가치관에 맞게 출력을 최적화하는 과정. ChatGPT 등 상업 모델의 핵심 비결.

  • 단점: 복잡한 3단계 파이프라인, 보상 모델 학습 추가 필요, 막대한 연산 비용

DPO — 직접 선호도 최적화

RLHF의 문제

보상 모델을 별도로 학습해야 함. 파이프라인 복잡, 불안정하고 느림

DPO의 해결

보상 모델 없이 선호도 데이터를 직접 손실함수로 최적화. 단순하고 안정적

  • 선택된(chosen) 응답과 거부된(rejected) 응답 쌍만 있으면 학습 가능
  • RLHF 대비 학습 안정성 높고 구현이 간소화됨

14. 상황별 최적 방법 선택 가이드

When to Use Which Method — Practical Decision Guide

OPTION 01

프롬프트 엔지니어링

학습 없이 입력 텍스트만 최적화. 간단한 형식 교정이나 일반 태스크에서 가장 빠르고 비용 제로. 항상 첫 번째로 시도.
형식 변경일회성 태스크데이터 없을 때
모델의 근본 행동·도메인 지식은 변경 불가
OPTION 02

RAG (검색 증강 생성)

외부 데이터·사내 문서 접근 권한 부여. 최신 정보나 기밀 문서에 강함. 모델 행동 자체는 변경 불가.
실시간 데이터사내 문서 QA지식 업데이트 빈번
행동·스타일은 변경 불가. 검색 품질에 전적으로 의존
OPTION 03 ★ 권장

SFT + LoRA / QLoRA

모델의 행동·어투·스타일 변경 또는 도메인 지식 주입. 자원 제약 시 첫 번째 선택. 대부분의 실무 파인 튜닝 시나리오에서 최선.
행동·스타일 변경도메인 지식 주입자원 제한빠른 실험
VRAM ≥24GB → LoRA, 부족 또는 30B+ → QLoRA
OPTION 04

SFT + DPO / RLHF

사용자 만족도, 안전성, 인간 선호도가 핵심인 서비스. ChatGPT류 어시스턴트를 구축할 때 필수. SFT 이후 추가로 적용.
챗봇 품질안전성 중요인간 선호도 최적화
DPO가 RLHF보다 구현 단순하고 성능 동등하거나 우월
